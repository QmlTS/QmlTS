# Step 6: V2 App Build Path — Design Spec

## Problem Statement

After Steps 1–5, the compiler emits V2-shaped QML and the native host supports V2
multi-instance routing. But the application **build path** — entry generation, output
layout, and QML tooling artifacts — is still entirely V1-shaped. The generated entry
calls `registerViewModel(className)`, no `qmldir`/`.qmltypes` files are produced, and
the output directory has no module structure. A V2 app cannot actually run from a
`qmlts build` output until this build path is upgraded.

## Scope

| In scope | Out of scope |
|---|---|
| V2 entry generation (`registerModule`) | User ViewModel class imports/factories in entry |
| qmldir generation per module | Third-party `qmlts.module.json` discovery |
| .qmltypes generation from schema | Library/package publishing |
| Module directory layout in output | Native host changes |
| ProductLayout & BuildManifest V2 fields | Compiler output changes |
| Build pipeline V2 threading | Dev server V2 hot-reload changes |
| Build-level diagnostics | qmllint V2 integration |
| Shared module metadata helpers | Runtime behavior changes |

## 1. Shared Module Metadata Helpers

All module-derived values must come from a single shared helper to prevent
entry generation, product layout, and manifest generation from drifting apart.

### 1.1 New Module: `src/build/module-meta.ts`

```typescript
export interface ModuleMeta {
  /** Full QML module URI (e.g. "TestApp.ViewModels"). */
  readonly moduleUri: string;
  /** Major version from config. */
  readonly versionMajor: number;
  /** Minor version from config. */
  readonly versionMinor: number;
  /** Human-readable version string (e.g. "1.0"). */
  readonly versionString: string;
  /** Directory path relative to QML output root (e.g. "TestApp/ViewModels"). */
  readonly moduleRelDir: string;
  /** qmltypes filename (e.g. "testapp_viewmodels.qmltypes"). */
  readonly qmltypesFilename: string;
  /** Sorted, deduplicated ViewModel type names. */
  readonly typeNames: readonly string[];
}

/**
 * Derive all module metadata from config and compilation schemas.
 * Returns undefined if there are no ViewModel schemas (no module to emit).
 */
export function deriveModuleMeta(
  moduleConfig: { prefix: string; version: { major: number; minor: number } },
  schemas: readonly { className: string }[],
): ModuleMeta | undefined;

/** Derive moduleUri from prefix. Always appends ".ViewModels". */
export function deriveModuleUri(prefix: string): string;

/** Module URI → output-relative directory path. Replaces all dots with "/". */
export function deriveModuleRelDir(moduleUri: string): string;

/** Module URI → qmltypes filename. Lowercases, replaces ALL dots with "_". */
export function deriveQmltypesFilename(moduleUri: string): string;
```

### 1.2 Derivation Rules

| Value | Rule |
|---|---|
| `moduleUri` | `prefix + ".ViewModels"` |
| `versionString` | `"${major}.${minor}"` |
| `moduleRelDir` | `moduleUri.replace(/\./g, '/')` — e.g. `"TestApp.ViewModels"` → `"TestApp/ViewModels"` |
| `qmltypesFilename` | `moduleUri.toLowerCase().replace(/\./g, '_') + '.qmltypes'` — global replacement of ALL dots. Example: `"Company.App.ViewModels"` → `"company_app_viewmodels.qmltypes"` |
| `typeNames` | Deduplicated (by `className`), sorted alphabetically. Returns `[]` if no schemas provided. |

### 1.3 No-ViewModel Case

If `runtime: "v2"` but the compilation result contains **no ViewModel schemas**,
`deriveModuleMeta()` returns `undefined`. The build pipeline must:
- **Not** emit a `registerModule()` call in the entry.
- **Not** create `qmldir` or `.qmltypes` files.
- **Not** create the module directory.
- Record `runtime: "v2"` in the manifest, but `modules` is `[]` (empty array).

This covers the case of a V2 app with only static QML views and no ViewModels.

### 1.4 Duplicate ViewModel Class Name Conflict (QMLTS-B002)

Multiple compilation units may produce schemas with the same `className` when
multiple views use the same ViewModel — the type name appears once in `typeNames`.

However, if two schemas share the same `className` but differ in their content
(different states, commands, effects, or lifecycle), this is a **build error**:

```
QMLTS-B002: Conflicting ViewModel schemas for "LoginViewModel" — multiple schemas
with the same className have different definitions
```

The build pipeline detects this by comparing schema contents (states, commands,
effects, lifecycle) for all schemas sharing a className. If they are structurally
identical, deduplicate. If they differ, emit `QMLTS-B002` and fail the build.

## 2. Entry Generator V2 Path

### 2.1 V2 Entry Code Shape

When `runtime: "v2"` and ViewModels are present, the entry generator produces:

```typescript
// Generated by QmlTS build system
// Do not edit manually

import { QmltsHost } from '@qmlts/host';
import { dirname, join } from 'node:path';
import { fileURLToPath } from 'node:url';

const distDir = dirname(fileURLToPath(import.meta.url));

const host = new QmltsHost({
  importPaths: [
    join(distDir, 'qml'),
  ],
});

// V2: Verify native host supports V2 runtime
if (typeof host.supportsV2 !== 'function' || !host.supportsV2()) {
  throw new Error(
    'This application requires V2 runtime support. ' +
    'Please update @qmlts/host to a version that includes V2 native bindings.'
  );
}

// V2: Register module types with QML engine
host.registerModule({
  moduleUri: 'TestApp.ViewModels',
  versionMajor: 1,
  versionMinor: 0,
  typeNames: ['CounterViewModel', 'LoginViewModel'],
});

host.loadFile(join(distDir, "qml/CounterView.qml"));

host.exec();
```

Key points:
- The `typeof host.supportsV2 !== 'function' || !host.supportsV2()` preflight
  gives a clear error if either the JS wrapper or native host is too old,
  instead of an opaque missing-function failure during `registerModule`.
- `host.registerModule(...)` uses the high-level object-form wrapper on `QmltsHost`.
  The native positional signature is internal to the wrapper.

### 2.2 V2 Entry Without ViewModels

When `runtime: "v2"` but no ViewModels exist (`moduleRegistration` is undefined),
the entry omits the `supportsV2()` check and `registerModule()` call entirely.
The rest of the V2 entry shape is identical (importPaths include `qml/`, etc.).

### 2.3 EntryGeneratorOptions V2 Extension

```typescript
export interface EntryGeneratorOptions {
  readonly compiledViewModels: readonly { readonly className: string }[];
  readonly mainQml: string;
  readonly qmlImportPaths: readonly string[];
  readonly packages?: ResolvedPackages;
  // V2 additions:
  readonly runtime?: 'v1' | 'v2';
  readonly moduleRegistration?: {
    readonly moduleUri: string;
    readonly versionMajor: number;
    readonly versionMinor: number;
    readonly typeNames: readonly string[];
  };
}
```

When `runtime` is `undefined` or `"v1"`, the entry generator produces the existing V1
output (byte-for-byte unchanged). When `runtime` is `"v2"` and `moduleRegistration`
is present, it emits the V2 code path with preflight + registerModule. When
`runtime` is `"v2"` and `moduleRegistration` is absent, it emits the V2 shape
without module registration (no-ViewModel case).

### 2.4 Fail-Fast

If `runtime: "v2"` but `moduleRegistration` is provided with an empty `typeNames`,
the entry generator throws. This is a programming error in the build pipeline —
the pipeline should pass `moduleRegistration: undefined` when there are no types,
not an empty registration block.

## 3. qmldir Generator

### 3.1 qmldir File Format

For each module, generate a standard Qt `qmldir`:

```
module TestApp.ViewModels
typeinfo testapp_viewmodels.qmltypes
```

The module types are registered programmatically via `registerModule()`, so
individual type declaration lines are not needed in the qmldir. The `typeinfo`
directive provides IDE/tooling support via the companion `.qmltypes` file.

### 3.2 Module URI → Directory Path

Derived by the shared helper `deriveModuleRelDir()`. Example:
`TestApp.ViewModels` → `TestApp/ViewModels/`.

### 3.3 qmltypes Filename Convention

Derived by the shared helper `deriveQmltypesFilename()`. Global replacement
of ALL dots with underscores, lowercased:
- `TestApp.ViewModels` → `testapp_viewmodels.qmltypes`
- `Company.App.ViewModels` → `company_app_viewmodels.qmltypes`

### 3.4 New Module: `src/build/qmldir-generator.ts`

```typescript
export interface QmldirGeneratorOptions {
  readonly moduleUri: string;
  readonly qmltypesFilename: string;
}

export interface QmldirGenerator {
  generate(options: QmldirGeneratorOptions): string;
}

export function createQmldirGenerator(): QmldirGenerator;
```

## 4. qmltypes Generator

### 4.1 qmltypes File Format

Follows the Qt `tooling 1.2` format. Includes only what the current native
host + compiler actually support. No state-change notify signals are emitted
unless they appear explicitly as `schema.effects` entries — the `.qmltypes`
file describes only the public schema contract, not internal Qt notify wiring.

Example using LoginViewModel:

```
import QtQuick.tooling 1.2

Module {
    Component {
        name: "LoginViewModel"
        prototype: "QObject"
        exports: ["TestApp.ViewModels/LoginViewModel 1.0"]
        exportMetaObjectRevisions: [0]

        Property { name: "username"; type: "string" }
        Property { name: "password"; type: "string" }
        Property { name: "isLoading"; type: "bool"; isReadonly: true }

        Method {
            name: "login"
        }

        Signal {
            name: "loginCompleted"
            Parameter { name: "success"; type: "bool" }
        }

        Method { name: "onMounted" }
    }
}
```

### 4.2 Schema → qmltypes Mapping

| Schema source | qmltypes output |
|---|---|
| `states[].qmlName`, `states[].qmlType` | `Property { name; type }`. If `readonly: true`, add `isReadonly: true`. |
| `commands[].qmlName`, `commands[].parameters` | `Method { name; Parameter[] }` |
| `effects[].qmlName`, `effects[].parameters` | `Signal { name; Parameter[] }` |
| `lifecycle.onMounted === true` | `Method { name: "onMounted" }` |
| `lifecycle.onUnmounting === true` | `Method { name: "onUnmounting" }` |

**Not emitted:** State-change notify signals (e.g. `usernameChanged`) are NOT
emitted as `Signal` entries. The Qt meta-object system handles NOTIFY signals
internally. Only signals explicitly declared in `schema.effects` appear as
`Signal` entries in `.qmltypes`.

**Parameter type mapping:** Schema parameters should usually carry QML type
strings, but the generator defensively maps known TypeScript primitives before
writing `.qmltypes`: `boolean` → `bool`, `number` → `real`, and
`unknown`/`any`/`object` → `var`. Existing QML type strings such as `string`,
`bool`, `int`, and `real` are preserved.

### 4.3 New Module: `src/build/qmltypes-generator.ts`

```typescript
  readonly moduleUri: string;
  readonly moduleVersion: { readonly major: number; readonly minor: number };
  readonly schemas: readonly ViewModelSchemaJson[];
}

  generate(options: QmltypesGeneratorOptions): string;
}

```

## 5. Product Layout V2 Extensions

### 5.1 ProductLayout Additions

```typescript
export interface ProductLayout {
  // ... existing fields unchanged ...
  /** V2: Module directory tree root (e.g., dist/qml/TestApp/ViewModels/). */
  readonly moduleDir?: string;
}
```

`moduleDir` is set only when `runtime: "v2"` and module config is present AND
there are ViewModel schemas. It is derived using the shared helper:
`join(layout.qmlDir, deriveModuleRelDir(deriveModuleUri(prefix)))`.

When there are no ViewModels, `moduleDir` remains `undefined` even for V2 builds.

### 5.2 materializeLayout

When `moduleDir` is present, create the directory tree during `materializeLayout()`.

### 5.3 BuildManifest V2 Additions

```typescript
export interface BuildManifest {
  // ... existing fields unchanged ...
  /** V2: Runtime version used for this build. */
  readonly runtime?: 'v1' | 'v2';
  /** V2: Module declarations produced by this build. Empty array if no ViewModels. */
  readonly modules?: readonly {
    readonly uri: string;
    readonly version: string;
    readonly types: readonly string[];
    /** Output-relative path, e.g. "./qml/TestApp/ViewModels/qmldir". */
    readonly qmldir: string;
    /** Output-relative path, e.g. "./qml/TestApp/ViewModels/testapp_viewmodels.qmltypes". */
    readonly qmltypes: string;
  }[];
}
```

**Portable paths:** `qmldir` and `qmltypes` paths in the manifest are
output-relative (prefixed with `./`), not absolute filesystem paths. This
ensures manifests are portable across machines and CI environments.

For V2 builds with no ViewModels, `runtime` is `"v2"` and `modules` is `[]`.

## 6. Build Pipeline Integration

### 6.1 phaseWriteOutput V2 Path

After writing compilation units and event bindings (unchanged), the V2 path:

1. **Derive module metadata** using the shared `deriveModuleMeta()` helper.
   If it returns `undefined` (no ViewModels), skip steps 2–3.

2. **Validate schema consistency.** Check for `QMLTS-B002` (conflicting schemas
   with same className). If detected, fail the build.

3. **Generate qmldir + qmltypes** and write to `moduleDir`:
   - `qml/TestApp/ViewModels/qmldir`
   - `qml/TestApp/ViewModels/testapp_viewmodels.qmltypes`

4. **Generate V2 entry** by passing `moduleRegistration` (or `undefined` for
   no-ViewModel builds) to the entry generator.

5. **Create V2-aware manifest** with `runtime` and `modules` fields.
   Paths stored as output-relative (`./qml/...`).

### 6.2 Diagnostic: QMLTS-B001

If `runtime: "v2"` and the compilation result has schemas but `config.module`
is missing (despite passing validation), emit `QMLTS-B001` as an error:

```
QMLTS-B001: V2 build requires module configuration but none was provided
```

This should be unreachable in normal flow (config validator catches this),
but the build pipeline has a defensive check.

### 6.3 Diagnostic: QMLTS-B002

If two or more schemas share the same `className` but have structurally
different definitions, emit `QMLTS-B002` as an error:

```
QMLTS-B002: Conflicting ViewModel schemas for "LoginViewModel" — multiple
schemas with the same className have different definitions
```

This is a real build error, not a warning. The build pipeline checks this
before generating module artifacts.

## 7. V1 Compatibility

### 7.1 V1 Path Unchanged

When `runtime` is `"v1"` (or undefined/default):
- Entry generator produces identical V1 output
- No qmldir/qmltypes files generated
- No moduleDir in ProductLayout
- No modules or runtime in BuildManifest
- All existing tests continue to pass byte-for-byte
